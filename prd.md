## 1. 개요 (Summary)

채용담당자가 짧은 시간 안에 백엔드·AI 지원자의 **역량·문제해결력·성장 잠재력·연락 가능 여부**를 파악할 수 있도록,
부드러운 스크롤과 은은한 "시스템 모티프" 배경을 갖춘 **미니멀한 자기소개 원페이지**를 만든다.

핵심 메시지: *"이 신입 백엔드·AI 개발자는 누구이고, 무엇을 설계·구현할 수 있으며, 어떻게 연락하는가."*

> 콘텐츠는 **김병주 실데이터**(GitHub `rlaqudwn1/backend-portfolio` 분석 기반)로 채운다. 이력서 PDF·LinkedIn·프로필 사진·정량 지표 등 일부 자산은 미확보로 대기.

---

## 2. 배경 & 문제 정의

- 신입/취준 단계에서는 경력 대신 **문제해결 과정·설계 사고·기술 학습 능력**을 보여줘야 한다.
- 일반 이력서(PDF)만으로는 차별화가 어렵고, 채용담당자의 기억에 남기 힘들다.
- 텍스트 나열형 페이지는 끝까지 읽히지 않는다 → **스크롤 몰입감 + 시각적 절제**가 필요하다.
- 화려한 효과는 로딩·성능·가독성을 해칠 수 있고, **백엔드 직군에서는 프론트 기교 자체가 역량 증명이 아니다.** 따라서 연출은 "장식"이 아니라 **직무와 연결된 은유(시스템·네트워크·데이터 흐름)** 수준으로 절제한다.

## 3. 타겟 사용자 (Persona)

### 주 사용자: 백엔드 채용담당자 / 현업 면접관
- **목표**: 30초~2분 안에 "면접 볼 가치가 있는 사람인가"를 판단.
- **행동**: 모바일 또는 데스크톱에서 빠르게 스크롤, 핵심만 스캔, 마음에 들면 코드·이력서·연락처 확보.
- **니즈**:
  - 한눈에 들어오는 한 줄 소개와 핵심 기술 영역.
  - **신뢰할 수 있는 증거**: 무엇을 어떤 규모로 설계·구현했는지, 어떤 문제를 어떻게 해결했는지(아키텍처·트러블슈팅).
  - 깃허브 코드·README 등 검증 가능한 링크.
  - **즉시 연락 가능한 수단**(이메일/이력서/GitHub).
- **불만 요소**: 느린 로딩, 과한 애니메이션으로 정보 탐색 방해, 모바일 깨짐, 근거 없는 자기 평가(별점), 연락처 못 찾음.

### 부 사용자: 본인(운영자)
- 콘텐츠를 쉽게 추가/수정할 수 있어야 한다(프로젝트·스킬 데이터를 코드와 분리 지향).

## 4. 목표 & 성공 지표

### 제품 목표
1. 채용담당자가 **이탈 없이 끝까지 스크롤**하게 만든다.
2. **연락/이력서/GitHub 확보 행동**을 유도한다.
3. 백엔드 기술 역량과 문제해결 잠재력을 **근거 있게(신뢰감 있게)** 전달한다.

### 성공 지표

> **측정 정책:** v1에서는 방문자 분석 도구를 도입하지 않는다(§12 Out of Scope). 트래픽 기반 지표는 **v1에서 보류**하며, 측정 가능한 성능 지표만 유지한다.

| 지표 | 목표 | v1 측정 여부 |
| --- | --- | --- |
| 첫 화면(LCP) 로딩 | ≤ 2.5초 | ✅ 측정 (Lighthouse/로컬) |
| Lighthouse 성능 점수 | 90+ | ✅ 측정 (Lighthouse) |
| 모바일 정상 렌더 | 100% (주요 해상도) | ✅ 측정 (수동 QA) |
| 평균 스크롤 깊이 | 70% 이상 | ⏸ 보류 — 분석 도구 도입 시 |
| 연락처/이력서 CTA 클릭률 | 측정 후 개선 | ⏸ 보류 — 분석 도구 도입 시 |

## 6. 정보 구조 (IA) & 섹션 요구사항

원페이지 세로 스크롤 구성. 네비게이션 바로 각 섹션 이동 가능.

```
[ Nav (고정/반투명) ]
   1. Hero / 인트로
   2. About (자기소개)
   3. 기술 스택 & 사용 경험
   4. 포트폴리오 / 프로젝트 (심층 케이스 스터디)
   5. 연락처 (Contact)
[ Footer ]
```

### 6.1 Hero / 인트로
- **목적**: 첫인상 + 한 줄 정체성 전달.
- **포함**: 이름, 직무 한 줄(백엔드·AI), 캐치프레이즈, 신뢰 라인(부스트캠프 AI Tech 8 · RMSE 3위), 스크롤 인디케이터, CTA("프로젝트 보기"·"GitHub").
- **배경(시스템 모티프)**: 노드·엣지 네트워크 + 미세 모션. 배경에만, 저채도, 가독성 방해 금지. 모바일 경량/정지.

### 6.2 About (자기소개)
- 프로필(선택), 2~4문단 자기소개, 핵심 키워드 태그. **성장 과정·학습 태도·협업 경험** 중심.

### 6.3 기술 스택 & 사용 경험
- **표현 방식(확정)**: **사용 맥락 중심.** 각 기술에 "어디에/어떻게/어떤 규모로 썼는지" 한 줄.
  - 예: `Redis — 조회 API 캐싱으로 응답 ~40% 단축`.
- **분류(예시)**: Language / Framework / Database / Infra·DevOps / Tool.
- **별점·레벨 바·% 게이지 미사용**(근거 약함·과장 우려).

### 6.4 포트폴리오 / 프로젝트 (심층 케이스 스터디)
- 대표 프로젝트 **1~2개**를 케이스 스터디로 깊게.
- **프로젝트당**: 문제/배경 · 본인 역할 · 아키텍처/데이터 흐름 · 기술 선택 이유·트레이드오프 · 트러블슈팅 · 결과·배운 점 · 링크(GitHub/문서/배포).
- 데이터는 코드와 분리 지향(`projects` JS/JSON).
- **아키텍처/데이터 흐름**은 Mermaid로 작성 → **사전 SVG 렌더**(`diagrams/*.mmd` → `*.svg`)해 정적 임베드(`<img>`). 런타임 의존 없음, 모바일 가독성 위해 세로(TD) 단일형.

### 6.5 연락처 (Contact)
- **포함(v1 실데이터)**: 이메일(`mailto:`) · GitHub.
- **이력서 PDF · LinkedIn · 전화번호 비노출** — 소스에 부재(이력서·LinkedIn 미보유). 확보 시 복원.

### 6.6 Footer
- 저작권, 반복 연락 링크(이메일·GitHub), 맨 위로 이동.

## 7. 사용자 흐름 (User Flow)

> 주 액터: 백엔드 채용담당자. 핵심 전환: 이메일 연락 · GitHub 방문 · 이력서 PDF 다운로드.

### 7.1 흐름 다이어그램

```
[진입: 채용메일 / 이력서 QR / 검색 / 직접 URL]
        │
        ▼
   1. HERO  (정체성 + 시스템 배경 + 커서 잔상 + CTA)
        ├──────► [CTA 즉시 클릭] ─► 5.CONTACT / GitHub   (스킵 경로)
        ▼ (스크롤 / Nav 점프)
   2. ABOUT → 3. SKILLS → 4. PORTFOLIO
                              └─► [GitHub/배포/문서 링크] ─► 외부 탭 (검증)
        ▼
   5. CONTACT  (이메일·GitHub·LinkedIn·이력서 PDF) ← 최종 전환
        ▼
   6. FOOTER  (반복 연락 · 맨 위로)
```
Nav는 고정/반투명으로 어느 지점에서든 1~5 섹션 점프 가능(비선형 탐색 허용).

### 7.2 단계별 흐름 (+ 이탈 지점)

| # | 단계 | 사용자 행동 / 의도 | 전환·이탈 |
| --- | --- | --- | --- |
| 1 | Hero | 첫인상 판단, 스크롤 시작 | CTA 즉시 클릭 시 5번 점프 |
| 2 | About | 스캔 — "어떤 사람인가" | 지루하면 이탈 → 간결·스캔 가능하게 |
| 3 | Skills | 기술 적합성 확인 | 별점 없음 → 사용 맥락으로 신뢰 형성 |
| 4 | Portfolio | 정독, 링크 클릭 — "증거" | **외부 링크 = 검증 분기** (새 탭) |
| 5 | Contact | 연락/다운로드 | **최종 전환 지점** |
| 6 | Footer | 재탐색/종료 | 위로 돌아가 재검토 |

### 7.3 전환 목표

| 우선순위 | 전환 행동 | 위치 |
| --- | --- | --- |
| 1차 | 이메일 연락(`mailto`) | Hero · Contact · Footer |
| 1차 | 이력서 PDF 다운로드 | Hero · Contact |
| 2차 | GitHub 방문(코드 검증) | Hero · Portfolio · Contact |
| 2차 | LinkedIn 방문 | Contact |

### 7.4 엣지 케이스

- **모바일/터치**: 커서 잔상 비활성, 배경 경량/정지. 흐름은 동일(세로 스크롤).
- **`prefers-reduced-motion`**: 스크롤·인뷰·배경·잔상 축소/비활성, 콘텐츠 흐름 유지.
- **느린 네트워크**: 배경·파티클 지연, 핵심 텍스트·CTA 먼저(LCP 보호).
- **JS 비활성/실패**: 시맨틱 마크업으로 콘텐츠·연락 링크 도달 가능.
- **이력서/LinkedIn 미보유 시**: 해당 CTA 미노출(빈 링크 금지).

## 8. 인터랙션 & 모션 요구사항

- **부드러운 스크롤**: CSS `scroll-behavior: smooth` + Nav 점프. 인뷰는 `IntersectionObserver`.
- **스크롤 인앤뷰 애니메이션**: 각 섹션/케이스 블록이 뷰포트 진입 시 페이드·슬라이드 업.
- **시스템 모티프 배경**: 저채도·저대비 배경 한정, 가독성 방해 금지(2D Canvas 네트워크).
- **마우스 추적 잔상/파티클**: 커서를 따라오는 입자가 부드럽게 페이드아웃. 입자 수·수명 상한으로 성능 가드. `prefers-reduced-motion`·터치 기기에서 비활성.
- **모션 접근성**: `prefers-reduced-motion` 시 애니메이션·배경·잔상 비활성/축소.
- **성능 가드**: 배경·파티클은 화면 밖/탭 비활성 시 렌더 일시정지, 모바일 경량/정지.

## 9. 디자인 가이드 (Visual)

- **톤**: 미니멀·클린·라이트. 넓은 여백, 절제된 색.
- **컬러**: 화이트/오프화이트 베이스 + 뉴트럴 그레이 텍스트 + 포인트 컬러 1개(블루 계열 권장). *(확인 필요: 최종 포인트 컬러)*
- **타이포**: 가독성 높은 산세리프, 한글/영문 페어링, 큰 제목 + 충분한 행간. *(확인 필요: 폰트 지정)*
- **그리드**: 일관된 컨테이너 폭, 8pt 스페이싱 시스템.
- **반응형**: 모바일 퍼스트, 데스크톱까지 우아하게 확장.

## 10. 기술 스택 & 아키텍처 (확정)

| 영역 | 결정 | 비고 |
| --- | --- | --- |
| 빌드/프레임워크 | **없음 — 순수 HTML/CSS/JS** | 의존성 최소, 가벼운 정적 뼈대 |
| 배경·커서 효과 | **2D Canvas** (경량 자체 구현) | 외부 CDN 의존 없이 시스템 모티프 + 잔상 |
| 부드러운 스크롤 | **CSS `scroll-behavior` + IntersectionObserver** | 프레임워크/라이브러리 미사용 |
| 스타일 | **순수 CSS** (CSS 변수로 토큰화) | 미니멀 톤 유지 |
| 콘텐츠 | 데이터 분리 지향(JSON/JS) | 현 단계는 시맨틱 HTML 직접 작성 |
| 배포 | *보류* | 추후 GitHub Pages 등 정적 호스팅 (§15) |

> 참고: 인접 폴더 `01-3dhomepage`에 Vite 스캐폴드가 있으나, 본 프로젝트는 의존성 최소화를 위해 순수 HTML/CSS/JS로 구성한다.

## 11. 비기능 요구사항 (Non-functional)

- **성능**: LCP ≤ 2.5s, 이미지 lazy-load, 스크립트 defer, Canvas 입자 수 상한.
- **반응형**: 모바일(≤480), 태블릿, 데스크톱 주요 해상도 정상 동작.
- **접근성(A11y)**: 시맨틱 마크업, 키보드 내비게이션, 색 대비 WCAG AA, `prefers-reduced-motion` 대응, 이미지 alt.
